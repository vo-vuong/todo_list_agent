## Current Work

Implement Phase 3: make Streamlit the primary demo interface with `Tasks` and `Planner` tabs that call the existing MCP server, planner, and report writer, while documenting and validating the final local demo flow.

## Files Likely To Change

- `pyproject.toml` - add Streamlit dependency and optionally a script or documented module launch command.
- `src/daily_planner_agent/mcp_client.py` - add MCP client helpers for all-task listing, adding tasks, and reset if the UI needs them.
- `src/daily_planner_agent/streamlit_app.py` - add two-tab Streamlit UI, action handlers, report preview, download, and clear error states.
- `tests/test_streamlit_app.py` - cover UI-safe helper behavior where feasible without launching a browser or making an LLM call.
- `tests/test_mcp_client.py` or existing tests - cover MCP payload extraction/helpers if expanded.
- `README.md` - update from Phase 2 CLI demo to final Streamlit primary UI, with server/UI commands and validation steps.

## Implementation Steps

1. Add Streamlit to the project and create `streamlit_app.py` with a small page shell containing exactly `Tasks` and `Planner` tabs.
2. Extend MCP client helpers as needed for UI task workflows: list all tasks, add task, complete task, reset tasks, and surface MCP connection/tool errors without masking user-action failures.
3. Build the `Tasks` tab with a task table, add-task form using the locked schema, completion selector/button, reset button, and refresh behavior after mutations.
4. Build shared UI error/status handling for the separate MCP server so the app tells the user when `uv run daily-planner-mcp` is not reachable at `MCP_SERVER_URL`.
5. Build the `Planner` tab so it checks missing `OPENAI_API_KEY`, fetches today's tasks through MCP, calls the existing PydanticAI planner, writes the markdown report, previews it, and offers a download button.
6. Update README with final setup, two-process run commands, Streamlit flow, missing-key behavior, reset behavior, and validation instructions using `uv`.
7. Add focused tests for helper code and run the existing suite, then perform a manual Streamlit smoke check if the local environment permits.

## Validation

- `uv sync`
- `uv run pytest`
- Start server: `uv run daily-planner-mcp`
- Start UI with the documented Streamlit command.
- In `Tasks`, list tasks, add a task, complete a selected task, and reset sample data.
- In `Planner`, with `OPENAI_API_KEY` set, generate and preview/download `reports/daily-plan-YYYY-MM-DD.md`.
- With `OPENAI_API_KEY` empty or absent, verify the Planner tab shows a clear error and does not generate an LLM plan.
- Stop or avoid starting the MCP server and verify the UI reports the MCP connection problem clearly.

## Risks And Constraints

- Streamlit callbacks around async MCP/planner helpers may need small synchronous wrapper functions to avoid event-loop issues.
- Full planner validation still depends on a real `OPENAI_API_KEY`; automated tests should avoid live LLM calls.
- The UI must not bypass MCP by reading or writing `data/tasks.json` directly for task actions.
- Existing CLI/MCP behavior must remain valid because README still supports command-line validation.
- Phase 2 has a non-blocking P2 finding about post-LLM task membership/priority validation. Phase 3 should not hide that risk; if the planner is touched, the small validation fix can be included, but UI delivery should stay focused.

## Out Of Scope

- Authentication, multi-user state, drag-and-drop schedule editing, browser persistence, notifications, and embedded single-process MCP.
- Replacing PydanticAI or FastMCP choices.
- Redesigning the generated markdown report format beyond preview/download presentation.
